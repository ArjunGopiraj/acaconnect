ALLOCATION SYSTEM IMPLEMENTATION GUIDE
======================================

This document outlines the complete approach used for implementing the venue allocation system,
which can be replicated for other allocation systems (volunteers, equipment, etc.).

OVERVIEW
========
The venue allocation system allows Hospitality team to allocate specific venues to published events,
and these allocations are automatically visible across ALL role-specific pages and dashboards.

IMPLEMENTATION APPROACH
======================

1. DATABASE SCHEMA ENHANCEMENT
------------------------------
File: backend/src/models/Events.js

Added hospitality object to Event schema:
```javascript
hospitality: {
  requirements_acknowledged: { type: Boolean, default: false },
  acknowledged_at: { type: Date },
  acknowledged_by: { type: mongoose.Schema.Types.ObjectId, ref: 'User' },
  allocated_rooms: [{
    room_number: String,
    room_name: String
  }],
  lab_allocated: String,
  venue_details: String,
  venue_allocated: { type: Boolean, default: false },
  venue_allocated_at: { type: Date }
}
```

Key Pattern: Create nested object for each allocation team with:
- acknowledgment tracking
- allocation data arrays/fields
- boolean flag for allocation status
- timestamps for audit trail

2. BACKEND CONTROLLER CREATION
------------------------------
File: backend/src/controllers/hospitality.controller.js

Created dedicated controller with methods:
- getEvents(): Fetch published events for allocation
- acknowledgeVenueRequirements(): Acknowledge requirements
- updateVenueAllocation(): Allocate venues with room details
- deleteVenueAllocation(): Remove allocation if needed

Key Pattern: Each allocation team needs:
- GET method to fetch events needing allocation
- POST method to acknowledge requirements
- PUT method to update allocation
- DELETE method to remove allocation

3. BACKEND ROUTES SETUP
-----------------------
File: backend/src/routes/hospitality.routes.js

Protected routes with role-based access:
```javascript
router.get('/events', auth, role(['HOSPITALITY']), getEvents);
router.post('/acknowledge/:eventId', auth, role(['HOSPITALITY']), acknowledgeVenueRequirements);
router.put('/allocate/:eventId', auth, role(['HOSPITALITY']), updateVenueAllocation);
router.delete('/allocate/:eventId', auth, role(['HOSPITALITY']), deleteVenueAllocation);
```

Key Pattern: Use role-based middleware to restrict access to specific team

4. FRONTEND DASHBOARD CREATION
------------------------------
File: frontend/src/dashboards/HospitalityDashboard.jsx

Dashboard with 4 main sections:
- Overview: Statistics and quick actions
- View Requirements: Events needing allocation
- Allocation Form: Dynamic form for allocation
- Allocated Items: View completed allocations

Key Pattern: Consistent dashboard structure with:
- Statistics overview
- Requirements viewing
- Allocation interface
- Completed allocations review

5. FRONTEND EVENTS PAGE CREATION
--------------------------------
File: frontend/src/pages/HospitalityEventsPage.jsx

Follows established design pattern:
- Header with navigation and user info
- Published events display
- Inline dashboard integration
- Consistent NIRAL 2026 theme

Key Pattern: Use existing events page as template for consistency

6. ALLOCATION DISPLAY ACROSS ALL PAGES
--------------------------------------
Updated ALL role-specific events pages:

Files Modified:
- frontend/src/pages/GeneralSecretaryEventsPage.jsx
- frontend/src/pages/TreasurerEventsPage.jsx
- frontend/src/pages/EventTeamEventsPage.jsx
- frontend/src/pages/ChairpersonEventsPage.jsx

Added venue display logic:
```javascript
{event.hospitality?.venue_allocated && (
  <div className="venue-details">
    <h4>📍 Venue Details</h4>
    <p>{event.hospitality.venue_details}</p>
    {event.hospitality.allocated_rooms?.length > 0 && (
      <div>
        <strong>Allocated Rooms:</strong>
        {event.hospitality.allocated_rooms.map((room, index) => (
          <span key={index}>{room.room_number} ({room.room_name})</span>
        ))}
      </div>
    )}
    {event.hospitality.lab_allocated && (
      <p><strong>Lab:</strong> {event.hospitality.lab_allocated}</p>
    )}
  </div>
)}
```

Key Pattern: Conditional rendering based on allocation status boolean

REPLICATION GUIDE FOR OTHER ALLOCATIONS
=======================================

To implement similar allocation system (e.g., Volunteer allocation):

STEP 1: DATABASE SCHEMA
-----------------------
Add to Event model:
```javascript
volunteers: {
  requirements_acknowledged: { type: Boolean, default: false },
  acknowledged_at: { type: Date },
  acknowledged_by: { type: mongoose.Schema.Types.ObjectId, ref: 'User' },
  allocated_volunteers: [{
    volunteer_id: { type: mongoose.Schema.Types.ObjectId, ref: 'User' },
    volunteer_name: String,
    role: String,
    contact: String
  }],
  volunteers_allocated: { type: Boolean, default: false },
  volunteers_allocated_at: { type: Date }
}
```

STEP 2: BACKEND IMPLEMENTATION
------------------------------
Create:
- backend/src/controllers/volunteer.controller.js
- backend/src/routes/volunteer.routes.js
- Add routes to server.js

STEP 3: FRONTEND DASHBOARD
--------------------------
Create:
- frontend/src/dashboards/VolunteerDashboard.jsx
- frontend/src/pages/VolunteerEventsPage.jsx

STEP 4: DISPLAY ACROSS ALL PAGES
--------------------------------
Add to ALL role-specific events pages:
```javascript
{event.volunteers?.volunteers_allocated && (
  <div className="volunteer-details">
    <h4>👥 Assigned Volunteers</h4>
    {event.volunteers.allocated_volunteers?.map((volunteer, index) => (
      <div key={index}>
        <strong>{volunteer.volunteer_name}</strong> - {volunteer.role}
        <br />Contact: {volunteer.contact}
      </div>
    ))}
  </div>
)}
```

STEP 5: ROLE SETUP
------------------
Create:
- backend/addVolunteerRole.js
- backend/addVolunteerUser.js

STEP 6: ROUTING
---------------
Add to App.jsx EventsRouter:
```javascript
case 'VOLUNTEER':
  return <VolunteerEventsPage />;
```

STEP 7: NOTIFICATIONS
--------------------
Add notification triggers in controller methods

KEY PRINCIPLES
==============

1. SINGLE SOURCE OF TRUTH
   - All allocation data stored in Event document
   - One database update affects all displays

2. CONDITIONAL RENDERING
   - Use boolean flags (venue_allocated, volunteers_allocated)
   - Only show allocation when confirmed

3. CONSISTENT PATTERNS
   - Same dashboard structure across all teams
   - Same display logic across all pages
   - Same role-based access control

4. AUDIT TRAIL
   - Track who allocated what and when
   - Maintain timestamps for accountability

5. NOTIFICATION FLOW
   - Notify relevant stakeholders of allocations
   - Keep workflow transparent

6. PROFESSIONAL UI
   - Consistent NIRAL 2026 theme
   - Responsive design
   - Proper error handling
